## Problem Statement

TransferMoneySaga (src/Account/Application/Saga/TransferMoneySaga.php) mixes
orchestration (DBAL transactions, repository calls, event dispatch) with
business rules (same-account check, currency match, amount currency match)
at lines 49-69. This is incorrect layering for a DDD reference implementation:
business rules belong in the Domain layer, not Application.

Phase 4 extracts these rules and introduces three DDD tactical patterns:
Domain Service, Specification Pattern, and Policy Pattern.

## What Makes This Cool

A single money transfer operation where all three patterns are visibly working:
Domain Service coordinates, Specification validates eligibility, Policy enforces
limits. A developer reading the code sees when to use each pattern and why they
exist as separate concepts. Clear enough for mid-level devs to learn from,
correct enough for seniors to respect.

## Constraints

- Symfony 7 / PHP 8.3
- DDD + CQRS + Event Sourcing + Hexagonal Architecture
- Account BC is fully event-sourced with projections
- Tests required for all changes (TDD approach)
- Domain Service must be pure domain (no infrastructure deps)
- Policy needs a port for external state (following Notification ACL pattern)
- Each pattern must be clearly identifiable as its own class hierarchy

## Premises

1. TransferMoneySaga mixes orchestration with business rules. Saga should become
   pure orchestrator: load accounts, call domain service, save, dispatch events.
2. Domain Service must collaborate with Policy and Specification in a single
   transfer flow. All three patterns visible in one operation.
3. Policy (TransferLimitPolicy) needs a real constraint (daily transfer limit),
   requiring transfer history tracking via a port/adapter.
4. Specification should be composable via and/or operators, demonstrating the
   pattern's core value: boolean composition of business rules.
5. ADR should document WHY these patterns exist and when NOT to use them.

## Cross-Model Perspective

Independent Claude subagent review. Key insights:

- Proposed "Policy-as-Specification" (unified interface). User chose separate
  hierarchies instead for pedagogical clarity. The teaching value of explicit
  labels ("this IS a Policy") outweighs interface unification.
- Confirmed TransferActivityPort should follow the Notification ACL pattern
  (Domain/Port/ with data DTOs). Consistent with existing conventions.
- Suggested two-PR split: refactor first (extract domain service), then
  additive feature (policy + specification). Good sequencing.
- Flagged: don't invent AccountActiveSpecification (no active/inactive concept
  in this model). Stick with real extracted rules.

## Approaches Considered

### Approach A: Separate Patterns (CHOSEN)
Three distinct class hierarchies. Domain Service, Policy, and Specification
as separate concepts. Domain Service explicitly calls Policy and Specification.
- Effort: M | Risk: Low | Completeness: 7/10
- Pro: Each pattern clearly labeled and identifiable
- Pro: Easy to point at "this is a Domain Service, this is a Policy"
- Con: More classes than unified approach

### Approach B: Policy-as-Specification (Unified Interface)
Policy implements SpecificationInterface. One composite spec, one call.
- Effort: M | Risk: Low | Completeness: 9/10
- Pro: Natural collaboration, cleaner Domain Service
- Con: Less obvious "here's the Policy pattern" label
- Rejected by user: pedagogical clarity > interface elegance

### Approach C: Domain Service Only (Minimal)
Extract rules into Domain Service only. Skip Specification and Policy.
- Effort: S | Risk: Low | Completeness: 4/10
- Rejected: doesn't deliver the three-pattern collaboration goal

## Recommended Approach

**Approach A** with separate pattern hierarchies.

### Domain Service flow:

```
MoneyTransferDomainService::validate(Account $from, Account $to, Money $amount)
    |
    +--> Specification composite:
    |      NotSameAccount AND CurrencyMatch AND AmountCurrencyMatch
    |      --> throws TransferValidationException if fails (reports which leaf failed)
    |
    +--> Policy check:
           TransferLimitPolicy::enforce($from->getId(), $amount)
           --> throws TransferLimitExceededException if fails
```

**Note on Domain Service purity:** `MoneyTransferDomainService` depends only on
domain interfaces (`SpecificationInterface`, `TransferLimitPolicyInterface`). The
`TransferLimitPolicy` implementation depends on `TransferActivityQuery` port interface
(also domain). Infrastructure adapters (`DoctrineTransferActivityQuery`) are injected
at runtime via DI. This is standard hexagonal architecture ... domain layer has zero
infrastructure imports.

### Refactored saga flow:

```
TransferMoneySaga::execute()
    |
    +--> Load accounts from repository
    +--> Call MoneyTransferDomainService::validate($from, $to, $amount)
    +--> Begin DBAL transaction
    |      Create Transaction (PENDING)
    |      $from->withdraw($amount)
    |      $to->deposit($amount)
    |      transaction->complete()
    |      Save all
    +--> Commit
    +--> Dispatch events
```

### Daily transfer limit design:

- Default limit: configurable via Symfony parameter (e.g. 10000.00 per day per account).
  Injected via constructor to `TransferLimitPolicy`. Testable by passing different values.
- TransferActivityQuery reads from `transaction` table via DBAL
  (sum of COMPLETED transfer amounts for account on given date; PENDING excluded)
- No new database table needed ... reuses existing transaction data
- Limit value injected via constructor (from Symfony parameters)

**TOCTOU race condition:** `validate()` runs before the DBAL transaction, so a
concurrent transfer could push the account over its daily limit between validation
and commit. Acceptable for a single-DB reference implementation. Production mitigation:
`SELECT ... FOR UPDATE` on the policy check or moving validation inside the transaction.
Documented in ADR as a known limitation.

## Implementation Details

### SpecificationInterface design:

```php
/**
 * @template T
 */
interface SpecificationInterface
{
    /** @param T $candidate */
    public function isSatisfiedBy(mixed $candidate): bool;

    public function and(self $other): self;
    public function or(self $other): self;
    public function not(): self;
}
```

PHPDoc `@template T` provides type safety via PHPStan without changing the runtime
interface. Leaf specs declare `@implements SpecificationInterface<TransferRequest>`.

Each leaf specification also implements a `reason(): string` method (not on the interface)
that the Domain Service uses to build error messages when `isSatisfiedBy()` returns false.
The composite `AndSpecification` iterates leaves and collects the first failing reason.

The `$candidate` is a DTO representing the transfer context:

```php
final readonly class TransferRequest
{
    public function __construct(
        public string $fromAccountId,
        public string $toAccountId,
        public Currency $fromCurrency,
        public Currency $toCurrency,
        public Money $amount,
    ) {}
}
```

Each leaf specification receives a `TransferRequest` and returns bool.

### TransferLimitPolicy design:

```php
interface TransferLimitPolicyInterface
{
    /** @throws TransferLimitExceededException */
    public function enforce(string $accountId, Money $amount): void;
}
```

Policy is NOT a specification. It's a separate pattern with a different
contract: `enforce()` throws on violation (imperative), while specifications
return bool (declarative). This distinction IS the teaching point.

### MoneyTransferDomainService design:

```php
class MoneyTransferDomainService
{
    public function __construct(
        private SpecificationInterface $transferSpecification,
        private TransferLimitPolicyInterface $transferLimitPolicy,
    ) {}

    public function validate(Account $from, Account $to, Money $amount): void
    {
        $request = new TransferRequest(
            $from->getId(), $to->getId(),
            $from->getCurrency(), $to->getCurrency(),
            $amount,
        );

        if (!$this->transferSpecification->isSatisfiedBy($request)) {
            throw TransferValidationException::fromSpecification(
                $request,
                $this->transferSpecification->reason(),
            );
        }

        $this->transferLimitPolicy->enforce($from->getId(), $amount);
    }
}
```

## Resolved Design Decisions

1. **Generics:** Yes, `@template T` PHPDoc generics on SpecificationInterface.
   PHPStan enforces type safety. Leaf specs declare `@implements SpecificationInterface<TransferRequest>`.
2. **Limit value:** Symfony parameter injected via constructor. More testable,
   matches existing patterns (e.g. configurable values in services.yaml).
3. **Failed spec reporting:** Yes. Leaf specs implement `reason(): string`.
   `AndSpecification` collects the first failing reason. Domain Service passes
   it to `TransferValidationException::fromSpecification()` for actionable errors.

## What I noticed about how you think

- You chose "make them collaborate" over isolated demonstrations. You want
  patterns to earn their existence by working together, not just existing
  as labeled files. That's the right instinct for a reference implementation.
- You picked "both" for the audience (mid-level + senior). You're not willing
  to sacrifice correctness for simplicity or vice versa. That's a hard bar
  to hit ... and it's the right one.
- You chose separate hierarchies over unified interface even when both models
  recommended unification. You valued pedagogical clarity over technical
  elegance. For a learning tool, that's the right call.
- "Mix of both" for quality bar, referencing specific existing patterns in your
  own codebase (Account aggregate style + Notification ACL style). You have a
  clear internal quality standard. That's taste.
